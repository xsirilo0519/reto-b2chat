# Order Manager

Sistema de gestión de pedidos de comercio electrónico construido con **Java 18 + Spring Boot 4 + WebFlux + R2DBC + RabbitMQ**.

---

## Tecnologías

| Tecnología | Uso |
|---|---|
| Java 18 | Lenguaje |
| Spring Boot 4.0.5 | Framework principal |
| Spring WebFlux | API reactiva no bloqueante |
| Spring Data R2DBC | Acceso reactivo a PostgreSQL |
| PostgreSQL | Base de datos relacional |
| Redis | Caché de productos (cache-aside) |
| Spring Data Redis Reactive | Integración reactiva con Redis |
| RabbitMQ | Mensajería asíncrona |
| Spring AMQP | Integración con RabbitMQ |
| JavaMailSender | Envío de emails |
| Spring Security 7 | Autenticación y autorización |
| JWT (jjwt 0.12) | Tokens Bearer para proteger endpoints |
| MapStruct | Mapeo de objetos |
| Lombok | Reducción de boilerplate |

---

## Arquitectura

El proyecto sigue **Clean Architecture** con módulos Gradle separados:

```
order-manager/
├── domain/
│   ├── model/          → Entidades, interfaces de gateways, excepciones
│   └── usecase/        → Lógica de negocio
├── infrastructure/
│   ├── driven-adapters/
│   │   ├── postgresql-repository/  → Implementación R2DBC
│   │   ├── rabbit-mq/              → RabbitMQ (publicar + escuchar + email)
│   │   └── redis-cache/            → Caché Redis reactivo (cache-aside)
│   └── entry-points/
│       └── reactive-web/           → Controllers REST
└── applications/
    └── app-service/                → Main, configuración de beans
```

### Schema de base de datos

```sql
CREATE TABLE users (
    id         BIGINT GENERATED BY DEFAULT AS IDENTITY (START WITH 1000 INCREMENT BY 1) PRIMARY KEY,
    name       VARCHAR(100)  NOT NULL,
    email      VARCHAR(150)  NOT NULL UNIQUE,
    address    TEXT          NOT NULL,
    created_at TIMESTAMP     DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP     DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE products (
    id             BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name           VARCHAR(150)  NOT NULL,
    description    TEXT,
    price          NUMERIC(10,2) NOT NULL CHECK (price >= 0),
    stock_quantity INT           NOT NULL CHECK (stock_quantity >= 0),
    created_at     TIMESTAMP     DEFAULT CURRENT_TIMESTAMP,
    updated_at     TIMESTAMP     DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE orders (
    id           BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id      BIGINT        NOT NULL,
    status       VARCHAR(20)   NOT NULL,
    total_amount NUMERIC(12,2) NOT NULL CHECK (total_amount >= 0),
    completed    BOOLEAN       NOT NULL DEFAULT FALSE,
    created_at   TIMESTAMP     DEFAULT CURRENT_TIMESTAMP,
    updated_at   TIMESTAMP     DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT fk_orders_user FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

CREATE TABLE order_items (
    id         BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    order_id   BIGINT        NOT NULL,
    product_id BIGINT        NOT NULL,
    quantity   INT           NOT NULL CHECK (quantity > 0),
    unit_price NUMERIC(10,2) NOT NULL CHECK (unit_price >= 0),
    total      NUMERIC(12,2) NOT NULL CHECK (total >= 0),
    CONSTRAINT fk_order_items_order   FOREIGN KEY (order_id)   REFERENCES orders(id)   ON DELETE CASCADE,
    CONSTRAINT fk_order_items_product FOREIGN KEY (product_id) REFERENCES products(id) ON DELETE RESTRICT
);
```

## Flujo de colas RabbitMQ

Este es el componente más importante del sistema. Toda la lógica de creación y procesamiento de pedidos ocurre de forma **asíncrona** a través de 5 colas.

### Exchange

Se usa un **TopicExchange** llamado `order.exchange`. Todas las colas se vinculan a él con su routing key.

### Mapa de colas

```
order.exchange
    │
    ├── order.process                ──▶  order.process.queue
    ├── order.processing             ──▶  order.processing.queue
    ├── order.notification.received  ──▶  order.notification.received.queue
    ├── order.notification.completed ──▶  order.notification.completed.queue
    └── order.notification.cancelled ──▶  order.notification.cancelled.queue
```

| Cola | Routing Key | Cuándo se publica | Quién escucha |
|---|---|---|---|
| `order.process.queue` | `order.process` | Al recibir `POST /orders` válido | `OrderProcessingListener` → crea orden PENDING |
| `order.processing.queue` | `order.processing` | Al recibir `PUT /orders/{id}/status` con PROCESSING | `OrderStockProcessingListener` → valida stock |
| `order.notification.received.queue` | `order.notification.received` | Cuando la orden se crea en PENDING | `OrderNotificationListener` → email PENDING |
| `order.notification.completed.queue` | `order.notification.completed` | Cuando la orden pasa a COMPLETED | `OrderNotificationListener` → email COMPLETED |
| `order.notification.cancelled.queue` | `order.notification.cancelled` | Cuando la orden pasa a CANCELLED | `OrderNotificationListener` → email CANCELLED |

### Flujo completo: Crear un pedido

```
1. Cliente → POST /orders
      │
      ▼
2. Valida usuario (404 si no existe)
   Valida productos (404 si no existe)
      │ ERROR → respuesta inmediata al cliente
      │ OK ↓
      ▼
3. Publica en order.process.queue
   Responde 202 ACCEPTED al cliente ⚡

      [En segundo plano]
      ▼
4. OrderProcessingListener recibe mensaje
   Calcula precios de cada ítem
   Crea orden en BD con status = PENDING
      │
      ▼
5. Publica en order.notification.received.queue
      │
      ▼
6. OrderNotificationListener envía email:
   "Tu pedido #N fue recibido"
```

### Flujo completo: Procesar un pedido (PROCESSING)

```
1. Cliente → PUT /orders/{id}/status  { "status": "PROCESSING" }
      │
      ▼
2. Actualiza orden en BD → status = PROCESSING
   Publica orden en order.processing.queue
   Responde 200 OK con orden en PROCESSING ⚡

      [En segundo plano]
      ▼
3. OrderStockProcessingListener recibe mensaje

      ¿order.completed = true?
      ┌──────────┬──────────┐
      SÍ         │         NO
      │          │          │
      │          │   Valida stock de cada ítem (transacción)
      │          │          │
      │          │    ┌─────┴──────┐
      │          │  Stock OK    Sin stock
      │          │    │             │
      │          │  Descuenta     Orden → CANCELLED
      │          │  inventario    Orden → CANCELLED
      │          │                Publica → order.notification.cancelled.queue
      │          │  Orden → COMPLETED
      │          │  Publica → order.notification.completed.queue
      │          │
      └──────────┘
   (ya fue completado antes, no descuenta de nuevo)
   Orden → COMPLETED
   Publica → order.notification.completed.queue

      ▼
4. OrderNotificationListener envía email:
   "Tu pedido #N fue completado"  ó  "Tu pedido #N fue cancelado"
```

### Campo `completed` en orders

El campo `completed BOOLEAN DEFAULT FALSE` protege contra el doble descuento de inventario:

| Escenario | `completed` | Acción al recibir PROCESSING |
|---|---|---|
| Primer procesamiento | `false` | Valida stock + descuenta + COMPLETED → `completed = true` |
| Re-procesamiento | `true` | Salta directo a COMPLETED sin tocar inventario |

## Autenticación JWT

Todos los endpoints (excepto `/auth/login`) están protegidos. Si se accede sin token o con uno inválido, se retorna **`401 Unauthorized`**.

### Flujo de autenticación

```
1. POST /auth/login  →  recibe el token JWT
2. Incluir en cada request el header:
       Authorization: Bearer <token>
```

### Credenciales

| Campo | Valor |
|---|---|
| `username` | `admin` |
| `password` | `admin` |

### `POST /auth/login` — Obtener token

**Request:**
```json
{
  "username": "admin",
  "password": "admin"
}
```

**Response `200 OK`:**
```json
{
  "token": "eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJhZG1pbiIs...",
  "type": "Bearer",
  "username": "admin"
}
```

**Errores:**
- `401 Unauthorized` — credenciales incorrectas

### Uso del token en los demás endpoints

Agregar el header en todas las peticiones protegidas:

```
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
```

**Ejemplo con curl:**
```bash
# 1. Obtener token
TOKEN=$(curl -s -X POST http://localhost:9090/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"admin"}' | jq -r '.token')

# 2. Usar el token en una petición protegida
curl -H "Authorization: Bearer $TOKEN" http://localhost:9090/products
```

### Configuración JWT en `application.properties`

| Propiedad | Descripción | Valor por defecto |
|---|---|---|
| `jwt.secret` | Clave secreta para firmar tokens (mín. 32 caracteres) | — |
| `jwt.expiration-ms` | Tiempo de expiración en milisegundos | `3600000` (1 hora) |

### Pedidos

#### `POST /orders` — Crear pedido *(asíncrono)*

Valida usuario y productos **antes** de publicar en RabbitMQ.

**Request:**
```json
{
  "userId": 1000,
  "items": [
    { "productId": 1, "quantity": 2 },
    { "productId": 2, "quantity": 1 }
  ]
}
```

**Response `202 Accepted`:**
```json
{
  "message": "Pedido recibido. Será procesado en breve."
}
```

**Errores (síncronos, antes de RabbitMQ):**
- `400` — campos inválidos
- `404` — usuario o producto no encontrado

> El usuario recibirá un email cuando la orden quede guardada en PENDING.

---

#### `GET /orders/{id}` — Obtener pedido

**Response `200 OK`:**
```json
{
  "id": 1,
  "userId": 1000,
  "status": "PENDING",
  "totalAmount": 5000.00,
  "completed": false,
  "items": [
    {
      "id": 1,
      "productId": 1,
      "quantity": 2,
      "unitPrice": 2500.00,
      "total": 5000.00
    }
  ],
  "createdAt": "2026-04-08T10:00:00",
  "updatedAt": "2026-04-08T10:00:00"
}
```

**Errores:**
- `404` — pedido no encontrado

---

#### `PUT /orders/{id}/status` — Actualizar estado *(PROCESSING es asíncrono)*

**Request:**
```json
{
  "status": "PROCESSING"
}
```

**Response `200 OK`** *(orden en PROCESSING inmediatamente)*:
```json
{
  "id": 1,
  "userId": 1000,
  "status": "PROCESSING",
  "totalAmount": 5000.00,
  "completed": false,
  "items": [...],
  "createdAt": "2026-04-08T10:00:00",
  "updatedAt": "2026-04-08T10:01:00"
}
```

> En segundo plano RabbitMQ valida el stock:
> - Si hay stock → orden pasa a **COMPLETED** + email al usuario
> - Si no hay stock → orden pasa a **CANCELLED** + email con motivo

**Otros valores válidos para `status`:**

| Valor | Descripción |
|---|---|
| `PENDING` | Estado inicial (asignado automáticamente) |
| `PROCESSING` | Dispara validación de stock async |
| `COMPLETED` | Asignado automáticamente si stock OK |
| `CANCELLED` | Asignado automáticamente si stock insuficiente |

**Errores:**
- `400` — valor de status inválido (responde con los valores permitidos)
- `404` — pedido no encontrado

## Emails enviados automáticamente

| Evento | Asunto | Cuándo |
|---|---|---|
| Orden recibida | `Pedido #N recibido` | Al crearse la orden en PENDING |
| Orden completada | `Pedido #N completado` | Al pasar a COMPLETED |
| Orden cancelada | `Pedido #N cancelado` | Al pasar a CANCELLED, incluye motivo |

## Manejo de errores

Todos los errores siguen el formato:

```json
{
  "status": 404,
  "message": "Recurso 'Producto' con id 99 no encontrado",
  "timestamp": "2026-04-08T19:43:29.5325225"
}
```

| HTTP | Causa |
|---|---|
| `400` | Validación de campos o enum inválido |
| `401` | Token ausente, inválido o expirado |
| `404` | Recurso no encontrado |
| `409` | Email de usuario duplicado |
| `500` | Error interno del servidor |
